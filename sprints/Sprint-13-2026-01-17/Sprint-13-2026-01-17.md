### What is done (high level)
- **Recruit sensitive notes**: `medicalNotes`, `dietaryRestrictions`, `extendedNotes` → `encryptedData` when user key exists (`lib/encryption/profileEncryption.ts`, recruit create/edit/detail).
- **Emergency contacts**: full sensitive payload → `encryptedData` when key exists; redacted placeholders for Firestore ordering (`lib/encryption/emergencyContactEncryption.ts`, recruit detail page).
- **Conversation message content (E2E path)**: per-message DEK, `encryptedPayload` + per-participant `keyWraps` on `conversations/{id}/messages/{id}`; thread page encrypts on send and decrypts for display/search; `POST /api/encryption/wrap-dek` wraps the DEK for the other participant (`lib/encryption/messageEncryption.ts`, `app/(dashboard)/conversations/[conversationId]/page.tsx`). **Plaintext fallback** if sender has no key or a participant has no `encryptionKeys` doc. **`lastMessageContent`** preview uses `"Encrypted message"` when the encrypted path is used.
- **Encryption APIs** (no dedicated settings page yet): `app/api/encryption/generate-key`, `key`, `recovery-code`, `recover-key`, `rotate-key`, **`wrap-dek`** (+ `lib/encryption/keyManager.ts`, etc.).
- **Browser compatibility logging**: `components/encryption/EncryptionCompatibilityCheck.tsx`.
- **One encryption status affordance**: recruit profile “encrypted at rest” note when `encryptedData` is present (`components/recruits/RecruitDetail.tsx`).

## Sprint Goal
Implement comprehensive end-to-end encryption for all sensitive data, including client-side encryption, encryption key management UI, key recovery system, encryption status indicators, encryption key rotation, encrypted data migration, and encryption compliance testing.

This includes client-side encryption before data storage, comprehensive key management, key recovery mechanisms, encryption status visibility, key rotation capabilities, data migration for existing unencrypted data, and compliance testing to ensure encryption meets security requirements.

## Notes

### Implementation Considerations
- **Messages (April 2026)**: `wrap-dek` learns the per-message DEK (not the message body) to produce the recipient’s wrap; Firestore stores ciphertext for the body when the encrypted path succeeds. For stricter models later, consider client-side asymmetric wrapping or conversation keys.
- Encryption is critical for GDPR compliance and data security
- Key management must be user-friendly but secure
- Key recovery must be secure and reliable
- Data migration must be carefully planned and tested
- Encryption performance must be acceptable

### Future Enhancements
- Advanced key management features
- Multi-device key synchronization
- Hardware security module (HSM) integration
